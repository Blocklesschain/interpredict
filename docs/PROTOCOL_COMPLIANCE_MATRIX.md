“Implemented and tested” means the requirement has direct assertions for its
material success and failure paths. A passing build, a nearby test, or the
presence of related code is not treated as proof.

## Cross-cutting findings that prevent a “complete” determination

1. **Test scope:** the suite now has 114 passing cases with broad negative,
   boundary, accounting, reputation, cancellation, and deterministic invariant
   coverage. Frontend/API/keeper integration, browser behavior, fuzzing, and
   formal verification remain untested; several cases deliberately prove defects.
2. **Snapshot integrity:** only an active-member count is snapshotted. The actual
   eligible electorate is evaluated at vote time, so post-snapshot additions and
   suspensions can change who votes without changing the quorum denominator.
3. **Solvency boundary:** standard-token property cases cover proposal seed,
   creator seed, trading collateral, fees, DEC rewards, refunds, multiple claim
   orders, cancellation, and treasury changes. The fee-on-transfer test is a
   concrete counterexample: accepted token behavior can make assets lower than
   liabilities, so the protocol is not unconditionally solvent.
4. **Reward/reputation semantics:** reward division counts all confirmed-outcome
   voters while accrual later requires current eligibility. Ineligible shares can
   remain stuck; recovered members can accrue suspension-period allocations; and
   incorrect voters can avoid penalties by never self-settling.
5. **Resolution escalation:** tied/no-quorum rounds reach `AdminVerification`, but
   controlled revoting and cancellation from that state are not implemented.
6. **Cancellation:** accounting and duplicate-refund behavior now have comprehensive
   tests, but there is no UI, reasons are unrestricted free text, required audit
   fields are absent, and `requestResolution` can win the state race and make
   cancellation unavailable.
7. **Token compatibility:** SafeERC20 is used, but fee-on-transfer/rebasing behavior
   is not rejected or balance-delta checked and can invalidate liability coverage.
8. **URI validation:** HTTPS is not enforced precisely and normal Arweave URIs fail
   the current fixed-width prefix comparison.
9. **Frontend/API/keeper completeness:** API lifecycle projection improved, but the
   live UI still omits creator/DEC detail, resolution/claims, search/sort, user
   positions, and most reward/quorum data. The corrected keeper route has no
   integration test and performs an O(n) sequential scan.
10. **V1 residue and regression risk:** multiple public documentation, governance,
    legal, lifecycle, simulator, and translation files still describe V1 methods,
    binary markets, native ITL economics, oracle settlement, or 5% fees.
11. **Established UI regression:** static comparison with `main` shows the V2
    homepage/dApp/Web3 context were reduced from 243/1,138/1,503 lines to
    37/200/50 lines. Branding fragments remain, but wallet session/chain handling,
    navigation targets, manual refresh, positions, resolution and claim flows,
    admin/team workflows, multilingual page content, and several accessibility
    behaviors were removed or not reimplemented. Browser-backed desktop/mobile
    verification remains pending because the browser-control backend was
    unavailable.
